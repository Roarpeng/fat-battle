## 一、项目概览

### 1.1 项目定位
**减肥大作战** 是一款面向上班族的健身游戏化 App，核心理念是"减肥是一个游戏活动，而不是靠毅力坚持的运动"。通过将每日饮食记录、运动打卡转化为与怪物的战斗系统，用 XP、成就、升级等游戏机制提供即时反馈快乐。

### 1.2 技术架构（双端并行）

| 维度 | Web 端 | Flutter App 端 |
|------|--------|----------------|
| 框架 | React 18 + Vite 5 | Flutter 3.x |
| 状态管理 | Zustand 5 + persist 中间件 | Riverpod 2.6 |
| 路由 | react-router-dom 7 | go_router 15 |
| 样式 | Tailwind CSS 4 + Framer Motion | Material Design |
| 构建 | Vite + PWA 插件 | Gradle (Android) |
| 存储 | localStorage (Zustand persist) | shared_preferences |
| 姿态识别 | MediaPipe Pose (CDN) | Google MLKit + tflite_flutter |
| 图表 | Recharts | fl_chart |

### 1.3 核心功能模块

```
web/src/
├── pages/           # 12 个页面（Battle/Food/Exercise/Pose/...）
├── components/      # 14 个组件（HpBar/BattleEffects/Toast/...）
├── store/           # 1 个 Zustand store（useGameStore.ts 28KB）
├── services/        # 4 个服务层（pose/foodRecognition/barcode/bluetooth）
├── data/            # 5 个数据文件（monsters/foods/exercises/taunts/encouragements）
├── lib/             # 2 个工具文件（gameAlgorithm/interaction）
└── types/           # 1 个类型定义文件（game.ts）
```

## 三、架构层完善建议

### 3.1 🔴 优先级：拆分巨型 Store（立即处理）

**现状**：`useGameStore.ts` 28KB，承载了用户状态、怪物状态、每日数据、成就系统、技能系统、物品系统、任务系统等全部状态。

**问题**：
- 任何一处修改都导致整个 store 的订阅者重渲染
- 代码冲突概率高，多人协作困难
- 持久化逻辑耦合在一起，容易出现 localStorage 溢出（5MB 限制）

**建议拆分方案**：

```
store/
├── useUserStore.ts       # 用户资料、BMI、设置
├── useBattleStore.ts     # 怪物、战斗、pendingAttack
├── useDailyStore.ts      # 每日摄入/消耗/记录
├── useProgressStore.ts   # 天数、连胜、历史
├── useAchievementStore.ts # 成就、任务、技能
├── useInventoryStore.ts  # 物品、金币、道具
└── stores.ts             # 统一导出 + 跨 store  action 编排
```

**参考实现**：使用 Zustand 的 `subscribeWithSelector` + 按 slice 拆分，保持接口兼容。

---

### 3.2 🔴 优先级：建立双端共享逻辑包（Monorepo）

**现状**：Web 端有 `gameAlgorithm.ts`，Flutter 端有 `game_algorithm.dart`，两份代码维护同一套公式（BMI、BMR、怪物 HP、伤害计算）。

**问题**：
- 项目记忆中明确提到"Game logic shared between App and mini-program via TypeScript package"，但实际并未实现
- 公式修改需要改两次，极易出现不一致

**建议**：

```
packages/
├── game-core/            # 纯 TypeScript 游戏逻辑包
│   ├── src/
│   │   ├── formulas/     # BMI/BMR/伤害/HP 计算
│   │   ├── monsters/     # 怪物生成器 + 克制系统
│   │   ├── achievements/ # 成就条件判断
│   │   └── index.ts
│   ├── package.json
│   └── tsconfig.json
```

- Web 端直接引用 `packages/game-core`
- Flutter 端通过 `flutter_js` 或将其编译为 WASM/JSON 配置来使用
- 至少将**怪物配置数据**和**成就定义**导出为 JSON，供双端共享

---

### 3.3 🟡 优先级：统一类型定义中心

**现状**：
- `types/game.ts` 定义了 `Monster`（有 `type/currentHp/isDefeated`）
- `store/useGameStore.ts` 定义了 `MonsterState`（有 `defId/tier/phaseIndex/isEnraged`）
- 两者描述的是同一概念，但字段不完全一致

**建议**：
1. 将 `MonsterState` 作为运行时状态，继承/组合自 `Monster` 定义
2. 或统一为单一 `Monster` 类型，废弃旧版字段
3. 在 `types/` 下建立分层：
   - `domain/` — 业务实体（Monster/Food/Exercise）
   - `dto/` — API/存储传输对象
   - `store/` — 前端状态形状

## 四、代码质量完善建议

### 4.1 🔴 引入单元测试（核心算法保护）

**必须测试的模块**：

| 模块 | 测试重点 | 工具建议 |
|------|----------|----------|
| `gameAlgorithm.ts` | BMI/BMR/怪物HP/伤害公式边界值 | Vitest |
| `monsters.ts` | 克制关系、阶段切换、狂暴触发 | Vitest |
| `useGameStore` actions | 攻击后怪物HP、金币计算、每日结算 | Vitest + @testing-library/react |
| `poseService` | 角度计算、重复计数逻辑 | Vitest |

**快速接入**：
```bash
npm install -D vitest @testing-library/react jsdom
```

### 4.2 🟡 引入 ESLint + Prettier

**现状**：`package.json` scripts 只有 `tsc --noEmit`，无静态代码检查。

**建议配置**：
```bash
npm install -D eslint prettier eslint-plugin-react-hooks eslint-plugin-react-refresh @typescript-eslint/parser @typescript-eslint/eslint-plugin
```

**关键规则**：
- `react-hooks/exhaustive-deps` — 防止 useEffect 依赖遗漏（BattlePage 中已有多个 timer 依赖问题）
- `react-refresh/only-export-components` — 配合 Vite HMR

### 4.3 🟡 添加 Error Boundary

**现状**：React 应用没有错误边界，一旦 poseService 初始化失败或 MediaPipe 加载异常，整个应用白屏。

**建议**：
- 在 `App.tsx` 外层包裹 `<ErrorBoundary>`
- 对 `PoseDetectionPage` 单独添加边界（摄像头权限拒绝是常见错误）
- 提供"重置应用状态"的降级 UI

## 五、性能优化建议

### 5.1 🟡 Store 订阅粒度优化

**现状**：
```tsx
const { monster, daily, coins, days, streak, user, setPendingAttack, attackMonster } = useGameStore()
```

BattlePage 订阅了整个 store，任何无关状态变化都会触发重渲染。

**建议**：
```tsx
// 使用 selector 精确订阅
const monster = useGameStore(s => s.monster)
const daily = useGameStore(s => s.daily)
const attackMonster = useGameStore(s => s.attackMonster)
```

或引入 `subscribeWithSelector` 中间件 + `shallow` 比较。

### 5.2 🟡 PWA 资源缓存优化

**现状**：`vite-plugin-pwa` 已配置，但：
- manifest 只有一个 SVG icon（iOS 不支持 SVG 作为桌面图标）
- 缺少 PNG 格式的多尺寸图标
- `workbox` 只缓存了 jsdelivr CDN，没有缓存 MediaPipe 模型

**建议**：
```js
// vite.config.ts 补充
manifest: {
  icons: [
    { src: '/icon-192.png', sizes: '192x192', type: 'image/png' },
    { src: '/icon-512.png', sizes: '512x512', type: 'image/png' },
  ]
},
workbox: {
  runtimeCaching: [
    // 缓存 MediaPipe 模型
    {
      urlPattern: /^https:\/\/cdn\.jsdelivr\.net\/npm\/@mediapipe\/pose\/.*/i,
      handler: 'CacheFirst',
      options: { cacheName: 'mediapipe-models', expiration: { maxAgeSeconds: 60*60*24*90 } }
    }
  ]
}
```

### 5.3 🟡 代码分割与懒加载

**现状**：所有页面在入口一次性加载。`BattlePage`、`PoseDetectionPage` 都包含大量逻辑和动画库。

**建议**：
```tsx
const PoseDetectionPage = lazy(() => import('./pages/PoseDetectionPage'))
const FoodPage = lazy(() => import('./pages/FoodPage'))
```

配合 `React.Suspense` 和统一的 `<PageSkeleton />` 组件。

## 六、产品体验完善建议

### 6.1 🔴 数据持久化安全

**现状**：Zustand persist 使用 `localStorage`，存在以下风险：
- 5MB 存储上限，长期数据（历史记录、成就）可能溢出
- 用户清缓存即丢失全部进度
- 无数据迁移机制（schema 变更后旧数据解析失败）

**建议**：
1. **分级存储**：
   - `localStorage`：用户资料、当前怪物、今日数据（高频小数据）
   - `IndexedDB`：历史记录、周统计数据（大量结构化数据）
2. **Schema 版本控制**：
   ```ts
   const STORAGE_VERSION = 3
   // 加载时检查版本，执行 migrate(v1 -> v2 -> v3)
   ```
3. **导出/导入备份**：提供 JSON 导出功能，让用户可手动备份

### 6.2 🟡 离线模式完善

**现状**：PWA 已配置，但：
- 食物识别服务依赖云端 API（离线不可用）
- 没有离线状态下的 fallback UI
- 没有网络状态检测

**建议**：
```tsx
// 使用 Navigator.onLine + online/offline 事件
const [isOnline] = useNetworkStatus()

// 食物识别离线 fallback
{!isOnline && (
  <div className="offline-banner">
    主人~ 离线模式下可以使用手动输入记录饮食哦 🐾
  </div>
)}
```

### 6.3 🟡 可访问性（a11y）

**现状**：
- 所有页面锁定 `max-w-[480px] mx-auto`，在平板/桌面端体验差
- 颜色对比度未验证（红色 `#FF6B6B` 在深色背景上可能不够）
- 缺少 `aria-label` 和键盘导航支持

**建议**：
1. 桌面端添加居中卡片式布局（而非单纯居中截断）
2. 使用 `@media (min-width: 768px)` 添加桌面适配
3. 关键按钮添加 `aria-label` 和 `:focus-visible` 样式

## 七、安全与健壮性建议

### 7.1 🟡 输入校验

**现状**：`useGameStore` 中的 `attackMonster`、`recordFood` 等 action 缺少参数校验。

**建议**：引入轻量级校验库（如 `zod`）：
```ts
import { z } from 'zod'

const AttackPayload = z.object({
  damage: z.number().min(0).max(10000),
  exerciseId: z.string().optional(),
})
```

### 7.2 🟡 防止状态篡改

**现状**：localStorage 中的数据可以被用户手动修改（改金币、改怪物 HP）。

**建议**：
- 对关键数据进行简单签名（checksum）
- 或至少对 `totalXp`、`days` 等字段做合理性校验（如 days 不能倒退）

## 九、总结

**减肥大作战** 是一款产品思路清晰、游戏化设计用心的项目。BOSS 设计系统（克制关系、多阶段、狂暴、赛季限定）已经达到了较高的完成度，UI 动效和交互细节也打磨得很到位。

**当前最核心的技术债务**在于：
1. **巨型 Store** 导致的维护性和性能问题
2. **双端逻辑重复** 带来的不一致风险
3. **缺乏测试和代码规范** 导致的回归风险

解决这三个问题后，项目的代码质量和可扩展性将有质的飞跃，后续添加社交功能、排行榜、多人协作等高级玩法也会更加顺畅。
