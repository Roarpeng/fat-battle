# 减肥大作战 — 产品需求文档

> 文档版本: v2.0 | 更新日期: 2026-06-19 | 产品阶段: 0-to-1 MVP

---

## 1. 需求背景

### 1.1 业务背景

中国超重/肥胖人群比例已超过50%，减肥健康类App市场规模超百亿。然而，传统减肥App的核心问题是**用户留存率极低**——超过70%的用户在下载后3天内放弃使用。根本原因在于：这些产品本质上是"数据记录工具"，用户每天打开App只是为了填写数字，过程枯燥、反馈滞后、缺乏持续动力。

Nintendo Switch《健身环大冒险》证明了游戏化可以大幅提升健身坚持率。但在饮食控制和日常减重领域，游戏化产品几乎空白。将减肥过程转化为RPG冒险，让用户通过控制饮食和锻炼来"击败脂肪怪"，有望解决动力缺失的核心痛点。

### 1.2 用户痛点

| 痛点 | 具体表现 | 后果 |
|------|---------|------|
| 缺乏动力 | 记录数据像完成任务，没有即时正向反馈 | 3天内放弃 |
| 枯燥乏味 | 每天重复相同的记录操作，毫无乐趣 | 打开率持续下降 |
| 反馈滞后 | 体重变化以周为单位，看不到即时成果 | 容易中途放弃 |
| 容易反弹 | 达到目标后缺乏维护机制，很快回到原点 | 减肥失败循环 |
| 动作不标准 | 自行锻炼不知道动作是否正确 | 效果差甚至受伤 |
| 饮食盲区 | 不知道食物具体热量，凭感觉吃 | 隐形热量超标 |
| 社交焦虑 | 排行榜上和大体重差距大，挫败感强 | 放弃使用 |
| 提醒骚扰 | 通知太多，开会/深夜时尴尬 | 直接关闭通知或卸载 |

### 1.3 不做这个产品的后果

用户继续使用传统减肥App，平均使用周期不超过7天，减肥成功率维持在5%以下。肥胖相关疾病（糖尿病、心血管疾病）风险持续上升，社会医疗负担加重。

## 2. 产品定位与目标

### 2.1 产品定位

一款**游戏化减肥App**，将用户的减肥历程设计成RPG冒险关卡。核心差异化：不是"记录工具"，而是"冒险游戏"。

### 2.2 核心目标

- 将用户7日留存率从行业平均15%提升至50%+
- 将减肥成功率从5%提升至30%+
- 让用户在愉悦中养成健康饮食和规律锻炼的习惯

### 2.3 目标用户

| 用户群体 | 特征 | 使用场景 | 痛点严重程度 |
|----------|------|---------|-------------|
| 上班族白领 | 25-35岁，久坐办公，外卖为主 | 工作日记录三餐，下班后锻炼 | 高 |
| 大学生 | 18-25岁，喜欢游戏化，预算有限 | 食堂/外卖饮食，宿舍/操场锻炼 | 中高 |
| 产后妈妈 | 28-40岁，需要科学减重，时间碎片化 | 居家饮食记录，碎片时间锻炼 | 高 |

## 3. 功能需求

### 3.1 功能总览

| # | 模块 | 功能描述 |
|---|------|---------|
| 1 | 角色创建 | 输入身高体重，计算BMI，选择角色形象，填写生活习惯问卷，进行体能评估，选择闯关难度，生成专属初始关卡 |
| 2 | 智能提醒 | 基于用户作息，在餐饮时段前推送拍照提醒；基于办公方式推送站立/喝水/活动提醒；支持勿扰模式和场景感知 |
| 3 | 食物识别 | 摄像头拍摄食物，AI识别菜品并估算卡路里，支持分量调整和混合场景估算，给出饮食建议 |
| 4 | 锻炼系统 | 室内锻炼通过摄像头识别动作，语音引导纠正，支持无摄像头模式；室外锻炼通过手机传感器记录步数速度 |
| 5 | 怪物战斗 | 每日精英小怪，饮食摄入使怪物回血，锻炼攻击削减怪物血量，击败获得积分；支持赛季制和体重平滑趋势 |
| 6 | 成就激励 | 每日任务完成后播放鼓励语音（基于用户选择的角色），发放积分和成就徽章；金币可兑换虚拟装扮或公益捐赠 |
| 7 | 数据统计 | 减肥进度（7日移动平均）、BMI变化、击败怪物数、总伤害、本周概览、历史趋势 |
| 8 | 语音交互 | 全程语音播报提醒、鼓励、指导，支持语音指令操作；智能场景检测（蓝牙耳机/外放/静音） |
| 9 | 社交系统 | 排行榜支持BMI区间分组和仅自己可见模式；断签缓冲机制 |

#### 3.2.1 角色创建

**页面结构：**

```
┌─────────────────────────────────────┐
│  减肥大作战 — 角色创建                │
├─────────────────────────────────────┤
│                                     │
│  [Step 1/5] 基本信息                │
│  ┌──────────┐  ┌──────────┐        │
│  │ 身高(cm) │  │ 体重(kg) │        │
│  │   170    │  │    75    │        │
│  └──────────┘  └──────────┘        │
│  ┌──────────┐                      │
│  │目标体重  │                      │
│  │    65    │                      │
│  └──────────┘                      │
│                                     │
│  BMI: 26.0 (偏胖)                   │

│                                     │
│  [下一步]                           │
│                                     │
└─────────────────────────────────────┘
```

**业务逻辑：**
- 用户输入身高、当前体重、目标体重
- 系统自动计算BMI = 体重 / (身高/100)^2
- 根据BMI判断体型：偏瘦(<18.5) / 正常(18.5-23.9) / 偏胖(24-27.9) / 肥胖(>=28)
- 目标卡路里 = 体重(kg) × 24（基础代谢估算）
- 目标体重必须小于当前体重，否则提示错误

#### 3.2.2 智能提醒系统

**餐饮提醒：**

| 触发条件 | 推送内容 | 时间窗口 |
|----------|---------|---------|
| 接近早餐时间（基于用户作息-30分钟） | "早安勇士！别忘了记录早餐哦，拍一拍就知道热量啦~" | 用户设定早餐时间前30分钟 |
| 接近午餐时间 | "午饭时间到！记得拍照记录，让脂肪怪无处可藏！" | 12:00-13:00 |
| 接近晚餐时间 | "晚餐时间~ 拍一拍，看看今天能不能击败怪物！" | 18:00-19:00 |
| 长时间未记录 | "已经X小时没记录饮食啦，脂肪怪正在偷偷恢复血量！" | 超过4小时无记录 |

**办公健康提醒（基于用户确认的办公方式）：**

| 办公方式 | 提醒类型 | 频率 | 内容 |
|----------|---------|------|------|
| 久坐办公室 | 站立提醒 | 每小时 | "坐了一小时啦，站起来走走，脂肪怪最怕你动起来！" |
| 久坐办公室 | 喝水提醒 | 每2小时 | "该喝水啦！保持代谢，让脂肪怪无处藏身~" |
| 久坐办公室 | 眼保健操 | 每3小时 | "眼睛累了吗？起来远眺一下，顺便活动活动！" |
| 偶尔走动 | 站立提醒 | 每1.5小时 | "起来活动一下吧，别让脂肪怪有机可乘！" |
| 偶尔走动 | 喝水提醒 | 每2.5小时 | "补充水分，保持战斗力！" |
| 经常外出 | 补水提醒 | 每3小时 | "外出也要注意补水哦，代谢不能停！" |
| 经常外出 | 步数鼓励 | 实时 | "今天已经走了X步，继续加油，脂肪怪在颤抖！" |

**提醒规则：**
- 用户处于睡眠时段（基于作息类型）不推送任何提醒
- 用户正在锻炼时不推送站立提醒
- 用户已记录饮食后不重复推送餐饮提醒
- 所有提醒支持语音播报

**勿扰模式与场景感知：**

| 场景 | 系统行为 | 用户设置 |
|------|---------|---------|
| 用户开启勿扰模式 | 所有提醒转为静默通知（不弹窗、不语音），仅记录待查看 | 可设置时段（如工作时间9:00-18:00） |
| 连接蓝牙耳机 | 语音播报正常播放 | 自动检测 |
| 外放模式/静音模式 | 语音播报转为震动+文字通知 | 自动检测 |
| 深夜加班（手动标记） | 锻炼提醒暂停，仅保留饮水提醒 | 用户手动标记"加班中" |
| 出差/旅行（手动标记） | 提醒策略临时切换为"外出模式"，减少站立提醒，增加补水 | 用户手动标记 |

**业务逻辑：**

- AI模型识别菜品类型，估算基础卡路里
- 用户可选择分量（小份/中份/大份），系统自动调整卡路里
- 系统分析当前餐次的营养结构，给出建议
- 记录后更新今日总摄入，影响怪物血量

**分量调整机制：**

| 分量 | 系数 | 示例 |
|------|------|------|
| 小份 | × 0.7 | 米饭小份 ≈ 160 kcal |
| 中份 | × 1.0 | 米饭中份 ≈ 230 kcal |
| 大份 | × 1.3 | 米饭大份 ≈ 300 kcal |

- 自动累加卡路里，并根据锅底/酱料类型增加加成
- 给出针对性的替换建议

**建议规则：**
- 如果某餐热量超过日均目标的40%：提示"这餐热量偏高，建议减少主食或增加蔬菜"

- 如果蔬菜不足：提示"蔬菜不够哦，多吃点菜，脂肪怪最怕这个"
- 如果油脂过多：提示"这餐有点油腻，下次可以少放点油"

## 8. 风险与依赖

### 8.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 食物识别准确率不足 | 中 | 高 | MVP阶段以手动记录为主，AI识别为辅；提供分量调整和混合场景估算 |
| 动作识别实时性 | 中 | 中 | 端侧模型优化，降低精度换取速度；提供无摄像头模式兜底 |
| 传感器数据不准 | 低 | 中 | 多传感器融合校准，允许用户修正 |
| 语音播报场景误判 | 中 | 低 | 提供手动场景切换，用户可覆盖自动检测 |

### 8.2 外部依赖

- 食物识别AI模型（自研/第三方API）
- 动作识别SDK（MediaPipe等）
- TTS/ASR服务
- 推送服务（FCM/APNs）

### 8.3 合规风险

- 摄像头权限：需明确告知用户仅用于动作识别和食物拍照
- 健康数据：体重、BMI等数据加密存储，符合隐私法规
- 语音数据：语音指令音频不存储，仅用于实时识别
